User Guide

Creating an HDMA Memory UVM Regression

This page describes how to set up a UVM verification environment for an HDMA memory design and how to build a small, traceable regression around three tests: hdma_mem_initialization, hdma_mem_randomize, and hdma_mem_full_burst_trf.

Target Verify HDMA memory behavior through reset, constrained-random traffic, and maximum burst transfers.
Method Combine sequence-driven stimulus, scoreboard checks, assertions, and coverage-driven closure.
Outcome A repeatable regression that can be run by seed, by test, or as a full sanity suite.
3 regression tests

One directed test, one constrained-random test, and one stress test for full bursts.

4 core components

Agent, sequences, scoreboard, and functional coverage are the minimum implementation set.

Pass criteria

No protocol errors, no scoreboard mismatches, and required coverage bins reached.

Verification Target

What the UVM environment must verify

The design under test is an HDMA memory block. The environment should prove correct initialization, data integrity, burst handling, and predictable behavior under randomized traffic.

Recommended environment content

  • A transaction class that models HDMA memory requests and responses.
  • A sequencer and driver that generate legal traffic into the DUT interface.
  • A monitor that reconstructs observed transfers for checking and coverage.
  • A scoreboard that compares observed memory behavior against the reference model.
  • Assertions for reset handling, address alignment, burst size legality, and handshakes.
  • A functional coverage model for address ranges, burst lengths, data patterns, and corner cases.

Regression Content

Define the three required tests

Each test should extend a common base test so configuration, agent setup, scoreboard hookup, and reporting stay consistent across the suite.

hdma_mem_initialization

Purpose: verify reset behavior and post-reset default memory state.

  • Drive reset and bring the interface into a known idle state.
  • Read key address regions after reset and compare against the expected initialization pattern.
  • Check that no stale data, spurious writes, or protocol violations appear during startup.
Directed 1-3 seeds Reset focused

hdma_mem_randomize

Purpose: explore legal randomized traffic and catch data-path corner cases early.

  • Randomize addresses, transfer sizes, write data, and read/write mixes under legal constraints.
  • Use the scoreboard to match all reads against the reference model.
  • Collect coverage for address windows, access types, burst lengths, and boundary conditions.
Constrained random 10-50 seeds Coverage driver

hdma_mem_full_burst_trf

Purpose: stress the DUT with maximum-length burst transfers and sustained throughput.

  • Create back-to-back full burst transactions at legal addresses.
  • Check burst packing, address progression, data integrity, and end-of-burst handling.
  • Measure whether the DUT handles long transfers without drops, stalls, or ordering defects.
Stress 5-20 seeds Burst focused
Implementation Flow

Build the environment in a regression-friendly way

The goal is not only to run one test once. The goal is to create a clean structure where the same environment supports compile-once and run-many regression execution.

Create a shared base test

Put all common setup in hdma_mem_base_test: environment creation, virtual interface configuration, default sequence selection, scoreboard enablement, and common objections.

Keep stimulus in sequences, not in tests

Each test should mainly select or configure a sequence. This keeps regression maintenance simple and avoids copying traffic generation logic into multiple test classes.

Make checking automatic

The scoreboard and assertions should run in every test without manual enable switches. A regression only scales when pass and fail conditions are encoded in the testbench itself.

Separate smoke and coverage goals

Treat hdma_mem_initialization as a fast smoke check, then use hdma_mem_randomize and hdma_mem_full_burst_trf for broader confidence and coverage closure.

Example test registration

class hdma_mem_initialization_test extends hdma_mem_base_test;
  `uvm_component_utils(hdma_mem_initialization_test)

  function new(string name = "hdma_mem_initialization_test",
               uvm_component parent = null);
    super.new(name, parent);
  endfunction

  virtual function void build_phase(uvm_phase phase);
    super.build_phase(phase);
    set_type_override_by_type(hdma_mem_base_seq::get_type(),
                              hdma_mem_init_seq::get_type());
  endfunction
endclass

Execution

Create and run the regression

Use a repeatable run flow that logs the test name, seed, and result for every execution. That makes triage and reruns straightforward.

Step Action Why it matters
1 Create a compile target for the HDMA memory UVM testbench. Compile once to eliminate avoidable differences between tests.
2 Create a regression list containing the three tests. This becomes the source of truth for the sanity suite.
3 Assign seed count per test based on risk and runtime budget. Random tests need more seeds than deterministic reset checks.
4 Capture logs, coverage databases, and a summary report. Regression output must be reviewable after the run finishes.
5 Promote the suite only after failures are reproducible and triage-ready. Non-reproducible regressions waste time and hide real design bugs.

Example commands

# compile once
make compile

# run one test
make test TEST=hdma_mem_initialization SEED=1

# run constrained-random seeds
make test TEST=hdma_mem_randomize SEED=101
make test TEST=hdma_mem_randomize SEED=102

# run the sanity regression
make regress TESTLIST=regression/hdma_mem_regression.list

Recommended pass/fail rules

  • Fail on any UVM error, fatal, scoreboard mismatch, or protocol assertion failure.
  • Fail if a test times out before all planned sequences complete.
  • Pass only if required coverage points are sampled for the intended scenario.
  • Store the seed in the report so every failure can be replayed exactly.
Release Checklist

Before calling the regression ready

Technical checks

  • The HDMA agent supports reset, read, write, and burst transactions.
  • The scoreboard predicts memory contents for all legal accesses.
  • The monitor samples enough detail to debug address, data, and burst issues.
  • The coverage model includes initialization, random accesses, and full burst scenarios.

Regression checks

  • hdma_mem_initialization is fast and stable enough for smoke use.
  • hdma_mem_randomize has legal constraints and replayable seeds.
  • hdma_mem_full_burst_trf exercises maximum burst length and boundary behavior.
  • The regression summary clearly shows pass/fail status per test and per seed.
